## Interaction Contract: Retry-Prefill Flow

This is the primary design content of this phase. It defines the interaction sequence when a user clicks "Retry" on a failed task in the sidebar.

### Current Flow (Broken)

```
User clicks "Retry" on error task in sidebar
  -> sidebar renderer calls window.clawSidebar.retryTask(id)
  -> IPC: sidebar:task-retry { id }
  -> ipc-handlers.ts calls agentManager.retryTask(id)
  -> Task auto-resubmits with original screenshot/DOM/instruction
  -> Overlay textarea is never prefilled
  -> User has no opportunity to edit the instruction
```

### Target Flow (Fixed)

```
User clicks "Retry" on error task in sidebar
  -> sidebar renderer calls window.clawSidebar.retryTask(id)
  -> IPC: sidebar:task-retry { id }
  -> ipc-handlers.ts:
     1. Reads original instruction from agentManager.getTask(id)
     2. Dismisses the error task from sidebar
     3. Sends overlay:prefill-instruction IPC with instruction text
     4. Activates overlay (expands bounds to full window)
     5. Sends overlay:mode-change 'rect-idle' to enter selection mode
  -> Overlay renderer receives prefill:
     a. Sets textarea.value to the instruction text
     b. Dispatches 'input' event (triggers auto-expand + enables submit)
     c. Activates overlay surface (body.claw-overlay--active)
     d. Enters rect-idle selection mode via dispatch({ type: 'ACTIVATE_RECT' })
     e. Shows input bar (positioned at bottom-center or follows selection)
  -> User sees:
     - Overlay active with selection crosshair mode
     - Input bar visible with previous instruction prefilled
     - Submit button enabled (because textarea has content)
  -> User can:
     - Edit the prefilled instruction text
     - Draw a new selection rectangle (or click an element if they switch modes)
     - Submit the edited instruction through the normal submit flow
     - Press Escape to cancel (returns to inactive, instruction is lost)
```

### State Transitions

| Step | Overlay State Machine | Input Bar | Sidebar |
|------|----------------------|-----------|---------|
| Before retry | `inactive` | Hidden | Error task visible with Retry/Dismiss buttons |
| After retry click | `rect-idle` | Visible, textarea prefilled | Error task dismissed |
| User draws selection | `rect-drawing` -> `rect-committed` | Visible, textarea still has prefilled text | No change |
| User submits | `inactive` (after submit IPC) | Hidden | New task appears (queued -> sending -> editing) |
| User cancels (Escape) | `inactive` | Hidden | No task (original was dismissed) |

### Timing

| Event | Duration |
|-------|----------|
| Retry click to overlay activation | Immediate (single event loop tick) |
| Overlay expand animation | Per existing behavior: instant bounds change via `setOverlayActive()` |
| Input bar appear | 150ms ease-out (existing `.claw-input-bar--visible` transition) |
| Textarea auto-expand after prefill | Immediate (synchronous DOM update from `input` event dispatch) |

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Retry while overlay is already active | Overlay stays active; textarea content is replaced with prefilled instruction; selection is cleared (re-enter rect-idle) |
| Retry with empty instruction (should not happen) | Guard: if `task.instruction` is falsy, do not send prefill IPC; fall back to activating overlay with empty textarea |
| Task not found (race condition: task dismissed between click and IPC) | Guard: if `agentManager.getTask(id)` returns null/undefined, return early; no overlay activation |
| Multiple rapid retry clicks | First click dismisses task and activates prefill; subsequent clicks have no task to find (guard catches this) |
| User switches to element mode after prefill | Allowed -- prefill sets rect-idle as default, but toolbar mode switch works normally |

### Original Task Dismissal Strategy

The error task is dismissed from the sidebar **when the retry handler fires**, before activating the overlay. Rationale:

- The instruction text is preserved in the prefilled textarea (not lost).
- If the user cancels (Escape), they lose the task entry but the instruction was visible and the user chose to retry then cancel.
- This matches the simplest implementation and avoids complex "keep task until re-submission succeeds" state tracking.
- Acceptable for v1.0 per RESEARCH.md Open Question 2 recommendation.

## Overlay Enhancement: Prefill Handler

The existing prefill handler in `overlay.ts` (lines 687-695) sets textarea value but does **not** activate the overlay or show the input bar. The handler must be enhanced.

### Current Handler (Insufficient)

```
onPrefillInstruction -> set textarea value -> dispatch input event
```

### Required Handler (Complete)

```
onPrefillInstruction ->
  1. Set textarea value
  2. Dispatch input event (auto-expand + submit enable)
  3. Add body.claw-overlay--active class
  4. Dispatch { type: 'ACTIVATE_RECT' } to state machine
  5. Show input bar (add .claw-input-bar--visible class)
  6. Position input bar at default location (bottom-center of viewport)
```

Step 3 is needed because the main process sends `setOverlayActive()` which expands the WebContentsView bounds, but the overlay renderer also needs to set its own body class for the semi-transparent capture background.

Steps 4-6 ensure the user sees the selection mode active with the input bar ready. Without these, the textarea is prefilled but invisible (input bar hidden, overlay in inactive visual state).

### Input Bar Positioning After Prefill

When prefilled (no selection exists yet), the input bar should appear at a **default position**: horizontally centered, vertically at 60% of the viewport height. This gives the user room to draw a selection above while seeing the prefilled instruction below.

| Property | Value |
|----------|-------|
| Default X | `(window.innerWidth - inputBarWidth) / 2` (centered) |
| Default Y | `window.innerHeight * 0.6` (60% from top) |
| Repositioning | When user commits a selection, input bar repositions below selection bounds (existing behavior) |

|---------|-----------|------------------|--------------|
